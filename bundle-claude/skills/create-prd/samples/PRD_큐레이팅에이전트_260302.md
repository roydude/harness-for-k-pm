
## 1. 기획 개요

- **제품명:** 큐레이팅 에이전트
- **기획 배경**
    - 1) 인재상 채용에 대해선 공감, But '역량 기반 채용' 개념 인지 부족
        
        → 자연스럽게 가이드를 따르다 보면 결과적으로 '좋은 채용'을 했다는 효용을 느끼게 하는 **넛지** 전략
        
    - 2) 이미 제품 내 다양한 자동화 기능이 존재, But 러닝커브 + 정확한 사용 타이밍 인지 힘듦
        
        → 채용 흐름 속에서 **적시적소**에 기능 세팅+실행하도록 유도하는 **가이드** 제공
        
- **제품 정의**
    - **큐레이팅:** 로깅 데이터를 바탕으로 사용자가 해야 할 일을 적시적소에 제안하고, 승인 시 기존 에이전트(이동·실행·분석)를 활용하여 **업무를 대행하는 에이전트 활동의 총체적인 개념**

## 2. 목적 및 목표

- **목적**
    - 채용 비전문가도 '역량 중심 채용 프로세스(완검화 플로우)'를 손쉽게 완주하도록 돕는다.
    - 사용자의 운영 리소스를 최소화하여, 제품 사용의 효용감을 극대화한다.
- **핵심 지표** (≈ 코파일럿 활용률)
    - **채용 만족도**: 운영 만족 + 결과 만족 → 서베이
    - **에이전트 활용 수:** (사용자가 직접 기능을 찾지 않고) 에이전트 제안을 통해 실행한 횟수

## 3. 검증 가설

- **가설**: 채용 진행 중 큐레이팅에 도움을 받은 사용자는 채용 에이전트에 대한 만족도가 높을 것이다.

## 4. 핵심 사용자 경험

1. **시스템이 사용자의 현재 상태를 보고 "지금 뭘 해야 하는지" 체크리스트를 추천한다.**
    1. 로그 기반으로 “지금 상태”를 해석 → 체크리스트 추천
2. **사용자가 체크리스트를 선택했다면, 체크리스트를 완료할 때 까지의 세부 액션 가이드를 제공한다.**
    1. 로그 기반으로 “지금 어디까지 했는지” 해석 → 다음 세부 액션 추천

### 5.2. 추천 조건 → Agent가 체크리스트 매핑

Agent에게 제공할 Context

| Context | 설명 |
| --- | --- |
| 로그 | 사용자 최근 활동 로그 |
| 로그 설명 | 로그의 액션별 의미 설명 |
| 채용 현황 | 진행중 상태의 채용별 현황 |
| 일일 집계 | 어제 신규 지원자 수 등 전체 채용에 대한 일일 집계 |
| 사용자 설정 현황 | 기본 설정 완료 여부 등 |
| 체크리스트 Pool | 각 체크리스트의 trigger_condition |

### 5.3. 체크리스트의 ‘actions + pain_point’ 데이터 → Agent가 세부 액션 제안

Agent에게 제공할 Context

| Context | 설명 |
| --- | --- |
| 체크리스트 정의 | name, actions, pain_point |
| 로그 (save 등) | 저장, 발송 등 핵심 액션 로그만 |

**로그 범위:** 버튼 클릭 전체가 아니라, save 같은 **핵심 액션 로그만** 수집

→ 모달 열림 같은 건 로그로 안 잡고, Agent가 actions 순서 + save 로그 조합으로 판단

### 5.4. 세부 액션 가이드 → 완료 감지

| 액션 | 완료 감지 방식 |
| --- | --- |
| 시간 | N초 이상 딜레이 시 |
| 내용 저장 | **save 로그 감지** |

## 6. 사용자 시나리오

### 6.1. 채용 운영 중: 역검 전형 안내 설정 제안

(체크리스트 선택)

1. 채용 담당자 김가온이 오전 9시에 채용솔루션에 로그인한다.
2. 에이전트 홈 화면에 큐레이팅 제안이 보인다.
3. 첫 번째 항목을 보니 "2026년 상반기 공채(진행중) | 역검 전형 안내 설정을 확인하세요."라고 적혀있다.
4. 그 아래에는 "전형 안내 대상자 48명”이라는 수치가 보인다.
5. 김가온은 "안내 메일 설정? 대상자가 48명이나 있어? 해야 하는 건가?"라고 생각하며 카드를 클릭한다.

(실행 가이드 시작)

1. 클릭하자 플로우 화면으로 이동하고 “가이드를 시작할게요”라는 토스트 메시지가 보인다. 그리고 화면 우측에 **실행 가이드 패널**이 열린다.
2. 실행 가이드 패널 상단에 "역검 전형 안내 설정"라는 체크리스트 제목이 보이고, 그 아래에 **세부 액션 목록**이 표시된다:
    - ☐ 전형 안내 설정 열기 [열기]
    - ☐ 주요 요소 확인
        
        ㄴ 전형 안내 방법 - 어떤 수단으로 안내하는지?
        
        ㄴ 발송 시간 - 언제 발송되는지?z
        
        ㄴ 내용 - 매핑값이 잘 들어갔는지?
        
    - ☐ 내용 저장
    - ☐ 발송 대상자 확인
3. 사용자는 첫 번째 액션 "전형 안내 설정 열기" 옆의 **[열기]** 버튼이 보인다.

(세부 액션 실행 (1/4): 전형 안내 설정 열기))

1. 김가온이 [열기] 버튼을 클릭한다.
2. **바로 '지원자 전형 안내 메일 수정' 모달이 열린다.**
3. 실행 가이드에서 첫 번째 액션이 완료(✓) 처리되고, 두 번째 액션 "주요 요소 확인"이 하이라이트된다.

(세부 액션 실행 (2/4): 주요 요소 확인)

1. 두 번째 액션 "주요 요소 확인"이 펼쳐지며 **가이드**가 표시된다
    - 전형 안내 방법(채용사이트, 메일, SMS)
    - 전형 안내 게시 기간
    - 예약 발송 일시
    - 내용(매핑값)
2. 김가온은 모달을 보면서 각 요소를 확인한다.
3. 실행 가이드에서 "주요 요소 확인" 액션을 **직접 체크**하거나, 해당 영역을 **클릭/수정하면 자동** 완료(✓) 처리된다.

(세부 액션 실행 (3/4): 내용 저장)

1. 세 번째 액션 "내용 저장"이 하이라이트된다.
2. 실행 가이드에 "설정을 저장해주세요. 저장해도 바로 발송되지 않아요."라는 안내가 표시된다.
3. 김가온이 모달의 "저장" 버튼을 클릭한다.
4. 세 번째 액션이 완료(✓) 처리된다.

(세부 액션 실행 (4/4): 발송 대상자 확인)

1. 네 번째 액션 "발송 대상자 확인"이 하이라이트된다.
2. 실행 가이드에 "발송 대상자 48명을 확인해주세요. [대상자 보기]"라는 안내와 버튼이 표시된다.
3. 김가온이 [대상자 보기] 버튼을 클릭하니, [지원자 관리] 화면으로 이동한다.
4. 해당 화면에는 해당 전형 지원자 리스트가 보인다.
5. 네 번째 액션이 완료(✓) 처리된다.

(체크리스트 완료 및 다음 제안)

1. 실행 가이드에 "전형 안내 설정 완료! 예약된 시간에 48명에게 안내될 예정이에요."라는 완료 메시지가 표시된다.
2. 실행 가이드가 다음 체크리스트를 제안한다: **"다음으로, 응시 완료 독려 안내를 미리 설정해보시겠어요?"**
3. 김가온은 [다음 체크리스트 시작] 버튼을 클릭하거나, [나중에 하기]를 선택할 수 있다.

### 6.2. 채용 결과: 반기 채용 현황 집계

(체크리스트 선택)

1. 채용 담당자 김가온이 1월 2일 오전 10시에 채용솔루션에 로그인한다.
2. 에이전트 홈 화면에 큐레이팅 제안이 보인다.
3. 첫 번째 항목을 보니 "지난 반기 채용 현황을 정리해보세요"라고 적혀있다.
4. 그 아래에는 "2025년 하반기 채용 6건 완료"라는 수치가 보인다.
5. 김가온은 "맞다, 하반기 채용 정리해서 팀장님께 보고해야 하는데"라고 생각하며 카드를 클릭한다.

(실행 가이드 시작)

1. 클릭하자 [지원자 관리] 화면으로 이동하고 "가이드를 시작할게요"라는 토스트 메시지가 보인다. 그리고 화면 우측에 **실행 가이드 패널**이 열린다.
2. 실행 가이드 패널 상단에 "반기별 채용 현황 집계"라는 체크리스트 제목이 보이고, 그 아래에 **세부 액션 목록**이 표시된다:
    - ☐ 채용 현황 질의 [예시 질의 입력]
    - ☐ 결과 확인
    - ☐ 데이터 내보내기
3. 사용자는 첫 번째 액션 "채용 현황 질의 " 옆의 **[예시 질의 입력]** 버튼이 보인다.

(세부 액션 실행 (1/3): 채용 현황 질의)

1. 김가온이 [예시 질의 입력] 버튼을 클릭한다.
2. **채팅창 프롬프트에 "2025년 하반기(7월~12월) 전체 공고의 총 지원자 수, 최종 합격자 수, 전년 대비 증감을 알려줘"가 자동 입력된다.**
3. 김가온이 엔터를 누른다.
4. 첫 번째 액션이 완료(✓) 처리된다.

(세부 액션 실행 (2/3): 결과 확인)

1. 두 번째 액션 "결과 확인"이 하이라이트된다.
2. 에이전트가 결과를 응답한다.
3. **에이전트 응답이 완료되면, 실행 가이드 에이전트가 이를 감지하고 두 번째 액션이 자동 완료(✓) 처리된다.**

(세부 액션 실행 (3/3): 데이터 내보내기)

1. 세 번째 액션 "데이터 내보내기"가 하이라이트된다.
2. 실행 가이드에 "결과를 저장하세요. [엑셀 다운로드]"라는 안내와 버튼이 표시된다.
3. 김가온이 [엑셀 다운로드] 버튼을 클릭한다.
4. **채팅창 프롬프트에 "위 결과를 엑셀로 다운로드할게. 사유: 2025년 하반기 채용 현황 보고"가 자동 입력된다.**
5. 김가온이 엔터를 누른다.
6. 에이전트가 엑셀 파일을 생성하고 다운로드 링크를 제공한다.
7. 세 번째 액션이 완료(✓) 처리된다.

(체크리스트 완료 및 다음 제안)

1. 실행 가이드에 "반기별 채용 현황 집계 완료! 다음 채용 계획에 활용해보세요."라는 완료 메시지가 표시된다.
2. 해당 체크리스트가 완료(✓) 처리된다.
3. 실행 가이드가 다음 체크리스트를 제안한다: **"개별 채용의 상세 성과 리포트도 확인해보시겠어요?"**
4. 김가온은 [다음 체크리스트 시작] 버튼을 클릭하거나, [나중에 하기]를 선택할 수 있다.

### 7.1. 체크리스트

- **체크리스트의 개념**
    - '체크리스트'는 에이전트가 사용하는 도구(tool) 개념과 유사
    - 사용자 상황에 따라 에이전트가 적절한 체크리스트를 제안함
- **체크리스트 시나리오의 조건**
    - 단순 조회(정보 전달)에서 끝나는 시나리오 지양 (예: 발송 실패 메일 건수 확인)
    - 단순 조회를 넘어, 의미 있는 행동까지 이어져야 함 (예: 발송 실패 메일 건수 확인 후 → 발송 세팅)
- **체크리스트 관리**
    - 1) 시나리오 정의
        - 완검화 의도를 전달하기 위해 필요한 시나리오 정의
            
            (예: 스마트 스크리닝 기준 설정, 블라인드 채용)
            
        - 사용자가 불편을 느끼는 패턴 감지
            
            (예: 역검 전형 단계 직전인데, 전형 안내 설정을 한 번도 확인하지 않은 경우)
            
    - 2) 시나리오 업데이트
        - 체크리스트 구조에 맞춰 시나리오 업데이트
    - 3) 개발자 전달 → 도구(Tool)화

### 플로우 항목별 로그 데이터

| **컬럼명** | **설명** | **예시** |
| --- | --- | --- |
| log_id | 로그 고유 ID | LOG_001 |
| created_at | 로그 생성 일시 | 2026-01-21 09:15:00 |
| user_id | 사용자 ID | USER_001 |
| recruitment_id | 채용 ID | REC_001 |
| recruitment_name | 채용명 | 2026년 상반기 공채 |
| recruitment_type | 채용 유형 | 공채 |
| node | 노드 (전형 단계) | 역량검사 전형 |
| tab | 노드 내 탭 | 기본 정보 |
| action | 항목 (수행한 액션) | 전형명 |
| action_type | 액션 유형 | edit |
- **로깅 대상**: 사용자가 플로우 화면에서 수정한 경우에만 로깅
- **로깅 시점**: 저장 버튼 클릭 시, 이전 값과 비교하여 변경된 항목만 개별 로그로 기록
- **로그 단위**: 변경된 항목 1개 = 로그 1개

### 99.3. 메모

### 시스템이 해야 할 일

- 시스템이 해야 할 일을 보면 크게 두 가지가 핵심이야.
    - 사용자의 상태를 해석하고, 상황에 맞는 체크리스트를 제안하는 것
        - 너와 내가 얘기해서 만든 건 '채용 운영' 시 필요한 체크리스트였고,
        - 지금은 그게 '기본 설정, 생성, 운영, 결과 분석'까지 확장해야 하는 상황이야. (운영 다음엔 결과 분석이 가장 중요함)
    - 사용자가 본인이 진행할 체크리스트를 선택하면, 체크리스트를 완료하기 위한 세부 액션을 에이전트가 제안하는 것
        - 사용자가 실제 수행하는 액션에 맞춰 실시간으로 맞춤 액션을 제안하는 게 관건이야.
        - 사용자 입장에선 제안하는 것을 따라가다보니, 자연스레 체크리스트를 완료하게 되는 거지.
        - 기존에는 체크리스트를 클릭하면 다음 화면으로 가서 채팅창을 열어주고 첫 번째 메시지를 던지는 것에서 끝났는데, 지금은 진짜 완수하기 위한 절차를 하나하나 가이드하는 걸 목표하고 있어. 그러면 사용자는? 진짜 잘 몰라도 그냥 클릭 클릭 하다보니 완료하게 되는 거야. 마치 Tourist라는 기능이 체크리스트별로 맞춤화되어 제공되는 느낌이지.
- 1, 2번 모두 '실시간으로 쌓이는 로그와 그 로그를 실시간으로 활용하는 LLM'이 관건일 것 같아.
- 그래서 그 구조를 개념화하는 게 지금 해야 하는 핵심 포인트라고 생각해.

### 실시간 로그와 LLM의 구조를 개념화한다면?

- 너가 말한 개념이 정확해. 핵심적인 사용자 경험(시스템의 역할)은 저게 맞아.
- 지금 해야 할 것은 실시간 로그, LLM 구조를 개념화하는 거라고 생각해.
- 현재 아래 내용을 고민 중이었어
    - 체크리스트가 추천되려면, 로그를 해석하는 과정이 먼저 있어야 함
    - 그러면 Context로 무엇을 제공할 것인가?
        - 스케줄러 Context Layer: 채용 현황 정보, 지원자 현황 정보, 사용자 현황 정보
            - '지원자 현황 정보'는 '단일 채용 정보'를 넘어서, 해당 기업(고객사)에게 통틀어서 지원된 지원자의 정보를 스케줄링으로 업데이트하는 것
            - '사용자 현황 정보'는 채용, 지원자 정보를 넘어서 사용자가 기본적인 환경 설정을 어디까지 진행했는지와 같은 정보를 스케줄링으로 업데이트하는 것
        - 로그 Context Layer: 로그 + 로그의 액션 항목별 (= 로그가 무엇을 의미하는지 알 수 있는 데이터)
        - 체크리스트 Context Layer: 사용자가 어떤 체크리스트를 완료해서 이제 보지 않아도 되는지, 봐야하는지 판단하기 위한 Context
            - 이 체크리스트를 기존엔 '채용'에 종속시켜놨는데, 이걸 공통영역으로 확대할 필요도 있을 것 같음.
            - 매 채용을 운영할 때마다 봐야 하는 체크리스트가 있을 것이고, 환경 설정처럼 한번 완료하면 다시 보지 않아도 되는 체크리스트가 존재할 것
    - 이 내용을 바탕으로 AI가 어떤 맥락의 체크리스트를 추천해야 하는지 결과가 나올 것 같아. 근데 이게 한 번에 쭉 처리되는 게 효과적일까?
    - 혹은 채용 운영 시 체크리스트 등 여러 체크리스트를 병렬로 뽑고 그 중 가장 적절한 것을 선택하게 하는 게 나을까?
    - 이 고민을 하기 전에 에이전트의 목적이나 각 프로세스를 먼저 구조화하는 게 필요할까?

### 아이디어

- 시나리오 대분류(기본 설정, 생성, 채용 운영, 결과 분석) 중, 이제 세 파트가 남은 상황
    - 우선순위: 채용 운영 > 결과 분석 > 기본 설정 > 생성
- **사용자 시나리오**를 구체화할 거고, 이중 **결과 분석**에 대한 부분을 구체화할 계획 중
- 다만 결과 분석의 경우 사용자에게 단순히 분석(데이터)만 제공하면 안되고, 제공의 '명분'이 분명해야 함
- 사용자가 효용을 느끼는 흐름은 이와 같음:
    - 사용자가 수상시 채용을 준비 중이어서 이전 채용의 성과를 참고하고 싶을 때
    - (대부분 중견, 대기업 채용 담당자) 사용자가 채용을 완료하고 1~2주쯤 지나서 이제 채용에 대한 성과를 정리하려고 할 때
    - 상반기/하반기 공개 채용을 준비하려는 시점에 이전 채용의 성과와 방법 등을 요약해서 전달해 줄 때 (참고자료 전달)

### 채용 결과 분석

- 지금은 두 가지 시나리오를 먼저 잡을 거야.
    - 채용 종료 후 결과 보고 준비
    - 상반기, 하반기 시작 시 직전 반기의 총 지원자 수, 채용 인원 수 정리 (이 용도는? 채용 계획?)
- 이 두 개로 시나리오를 구체화시킬까 해.
- 이와 관련된 맥락을 전달해 줄게.
    - **'채용 성과 리포트' 기능 (1번 사진)**
        - '채용 (계획)'별로 채용 성과 리포트가 존재함
        - 유저가 원하는 위젯을 넣어 커스터마이징할 수 있는 성과 리포트
            - 위젯은 모집 정보(전체 지원자 학력 분포, 지원자 경력 분포, 유입 채널 분포, 전형별 응시율, 역량 정보, 합격자 요약 정보, 채용 단계별 퍼널)로 존재함
        - (시나리오의 끝) 사용자는 채용 성과 리포트를 본인의 희망대로 편집하고, PDF로 다운로드할 수 있음
    - **'지원자 관리' 기능 (2번 사진)**
        - (채용, 공고, 전형 등) 지원자에 대한 테이블을 볼 수 있음
        - 사용자는 우측 패널(챗방)을 통해 통계·분석 에이전트를 실행해서 요약 정보를 확인할 수 있음
        - (시나리오의 끝) 사용자는 본인이 궁금한 데이터를 물어보고 집계된 데이터를 확인할 수 있음 (테이블 복사 혹은 엑셀 다운로드도 가능)
- 전자의 경우 '채용 결과'를 알리는 거기 때문에 채용 성과 리포트를 사용하는 체크리스트가 나오면 되고, 후자의 경우 여러 채용의 결과를 집계하고 분석하는 것이기 때문에 '지원자 관리' 체크리스트가 나오면 될 것 같아.
- 지원자 관리 체크리스트(시나리오)의 경우, 실행 액션 가이드가 사용자에게 버튼 등을 제시하면, '(어떤 질의를 입력해야 할지 안내하는 걸 목표로) 사용자 프롬프트창에 질의 예시가 바로 입력되는 형태로 고려 중이야)
