## 작성 가이드

### 기획 개요 작성법

기획 배경은 단순 사실 나열이 아니라, **"현상 → 시사점/전략 방향"** 구조로 쓴다.

나쁜 예:
> "사용자들이 기능을 잘 모른다"

좋은 예:
> "이미 제품 내 다양한 자동화 기능이 존재, But 러닝커브 + 정확한 사용 타이밍 인지 힘듦
> → 채용 흐름 속에서 적시적소에 기능 세팅+실행하도록 유도하는 가이드 제공"

### 사용자 시나리오 작성법

시나리오는 **스토리텔링**으로 쓴다. 개발 스펙이 아니라 "사용자의 하루"를 그린다.

나쁜 예:
> "사용자가 버튼을 클릭하면 모달이 열린다."

좋은 예:
> "김가온이 [열기] 버튼을 클릭한다.
> 바로 '지원자 전형 안내 메일 수정' 모달이 열린다.
> 실행 가이드에서 첫 번째 액션이 완료(✓) 처리되고, 두 번째 액션이 하이라이트된다."

핵심 원칙:
- **페르소나에 이름을 붙인다** (김가온, 이수진 등)
- **사용자의 내면 생각을 "따옴표"로 표현한다** ("안내 메일 설정? 해야 하는 건가?")
- **시스템 반응과 사용자 행동을 번갈아 기술한다**
- **시나리오 내 단계를 소제목(괄호)으로 구분한다**: (체크리스트 선택) → (실행 가이드 시작) → (세부 액션 1/4) → ...

### 검증 가설 작성법

"~하면 ~할 것이다" 형태로 검증 가능한 문장을 쓴다.

나쁜 예:
> "이 기능이 좋을 것이다"

좋은 예:
> "채용 진행 중 큐레이팅에 도움을 받은 사용자는 채용 에이전트에 대한 만족도가 높을 것이다."

### 의사결정 필요사항 작성법

PRD에는 "확정된 것"만 있어야 한다고 생각하기 쉽지만,
실제로는 **"아직 결정 안 된 것"을 명시하는 것**이 리뷰의 질을 높인다.

좋은 예:
> - **용어 확정**: '큐레이팅'이란 단어를 계속 사용할지? 혹은 '코파일럿'처럼 다른 단어로 바꿀지?

### 번호 체계

계층적 번호를 부여하되, 부록은 `99.`로 시작하여 본문과 시각적으로 구분한다.
- `5.1.` → 대항목
- `5.1.1.` → 하위 상세
- `99.1.` → 부록 항목

### 피그마 이미지 삽입

피그마 스크린샷이 있는 경우, 해당 섹션에 직접 삽입한다.
이미지 아래에 반드시 핵심 UI 요소와 인터랙션에 대한 텍스트 설명을 추가한다.
이미지만으로는 스펙이 아니다 — 이미지는 이해를 돕는 보조 자료일 뿐이다.
피그마 링크가 있는 경우 링크도 함께 제공한다 (스크린샷은 시점 고정, 링크는 최신 반영).
