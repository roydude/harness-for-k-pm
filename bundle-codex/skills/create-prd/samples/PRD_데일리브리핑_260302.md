## 1. 기획 개요

- **제품명:** 데일리 브리핑 (Daily Briefing)
- **컨셉 레퍼런스:** ChatGPT Pulse + Automation
- **기획 배경**
  - 채용 담당자는 출근 후 가장 먼저 "어제 지원자가 몇 명 왔는지", "역검 응시율이 어떤지", "접수 마감이 언제인지"를 확인한다. 이 정보를 파악하지 못하면 당일 업무 우선순위를 세우기 힘들어진다. (전형 안내를 보내야 하는지, 면접 일정을 잡아야 하는지 판단이 늦어진다) → 매일 필요한 채용 현황을 출근과 동시에 한눈에 제공해야 하는 게 필요하다
  - 현재 인사이트 카드(메인 > 인사이트의 지원서 제출 완료율 등)는 '고정형 정보'라 사용자가 원하는 정보와 불일치하는 경우가 많다. 원하는 정보를 얻으려면 매번 동일한 프롬프트를 직접 입력하거나, 여러 화면을 돌아다니며 찾아야 한다. → 반복되는 질의를 '자동화(스케줄러)'하고, 결과를 한 곳에 모아서 제공하는 구조가 필요하다. (인사이트)
  - 채용 담당자 대부분이 채용 비전문가(인사팀 겸직 등)여서, 어떤 데이터를 봐야 하는지 자체를 모르는 경우가 많다. → 시스템이 '기본 템플릿(추천 프롬프트 4~5개)'을 먼저 제공하여, 사용자가 뭘 봐야 하는지부터 안내한다.
- **제품 정의**
  - **오토 스케줄링 (인프라 레이어):** 사용자가 원하는 질의(프롬프트)를 등록하면, 스케줄러가 정해진 시간에 자동 실행하여 Raw 결과를 저장하는 자동화 인프라. 데일리 브리핑뿐 아니라 향후 다른 소비처(주간 리포트, 알림 트리거 등)에도 활용 가능하다.
  - **데일리 브리핑 (소비 레이어):** 오토 스케줄링이 생성한 실행 결과를 수집하여, 인사이트 리포트 에이전트가 시그널 탐지 기반의 종합 MD 리포트로 합성하고 매일 제공하는 기능.

## 2. 목적 및 목표

- **목적**
  - 채용 담당자가 출근 후 **별도 조작 없이** 오늘의 채용 현황과 해야 할 일을 즉시 파악할 수 있게 한다.
  - 데일리 브리핑을 통해 **제품의 다른 기능(지원자 관리, 전형 안내 등)으로 자연스럽게 연결**하여 제품 전체의 활용도를 높인다.
  - 채용 비전문가도 "뭘 봐야 하는지"부터 안내받아, 데이터 기반 채용 운영 습관을 형성하게 한다.
- **핵심 지표**
  - **데일리 브리핑 사용율**: 전체 활성 사용자 중 데일리 브리핑을 주 3회 이상 확인하는 사용자 비율 → 로그 분석
  - **후속 기능 전환율**: 브리핑 확인 후 제품 내 다른 기능(지원자 관리, 전형 안내 등)으로 이동한 비율 → 로그 분석

---

## 3. 검증 가설

- **가설 1**: 데일리 브리핑을 주 3회 이상 확인하는 사용자는, 그렇지 않은 사용자 대비 제품 내 기능(지원자 관리, 채용 액션 등) 사용 빈도가 높을 것이다.
- **가설 2**: 기본 템플릿 프롬프트를 먼저 제공한 사용자는, 빈 상태에서 직접 프롬프트를 작성해야 하는 사용자보다 데일리 브리핑 정착율(4주 연속 사용)이 높을 것이다.

## 4. 핵심 사용자 경험

1. **기본 템플릿으로 즉시 시작**
  1. 최초 진입 시 시스템이 **추천 프롬프트 4~5개**를 기본 세팅해준다. 사용자는 별도 설정 없이 바로 리포트를 받아볼 수 있다. (예: 어제 지원자 수, 지원서 미제출자 수, 오늘 할 일 제안 등)
2. **인사이트 리포트 확인 → 후속 행동 연결**
  1. 에이전트가 프롬프트 실행 결과를 종합하여 **하나의 MD 리포트**로 작성한다. 고정된 카드/stat-block이 아닌, 에이전트가 문단별로 자유롭게 구성하는 형태다. 날마다 인사이트의 중점이 달라질 수 있으므로, 에이전트가 텍스트·볼드 수치·인라인 표 등을 자유롭게 조합한다.
  2. 리포트 본문 아래에 **추천 액션 버튼 영역**이 별도 UI로 분리된다. 각 버튼은 제품 내 관련 페이지(지원자 관리, 전형 안내 설정 등)로의 딥링크이며, 필(pill) 형태로 "~화면으로 이동합니다. >" 텍스트와 함께 표시된다.
  3. 프롬프트별 Raw Data는 아코디언으로 펼쳐볼 수 있다.
3. **서브챗으로 추가 질의**
  1. 리포트를 보다가 궁금한 게 있으면 바로 서브챗을 열어 추가 질문한다. 현재 페이지의 요약 정보가 자동 주입되어 맥락을 이어간다.
4. **프롬프트 커스터마이징**
  1. 기본 템플릿에 만족하지 못하면, 사용자가 직접 프롬프트를 추가/수정/삭제할 수 있다. 단, 이 기능은 기본 템플릿 경험 이후의 2차 경험이다.

## 5. 기술 구조

데일리 브리핑은 두 개의 독립된 서브시스템으로 구성된다. **오토 스케줄링**은 프롬프트를 등록하고 주기적으로 실행하는 인프라 레이어이고, **데일리 브리핑**은 그 실행 결과를 소비하여 인사이트 리포트를 생성하는 소비 레이어다. 두 레이어는 "실행 결과(Raw Data)" 를 매개로 느슨하게 연결된다.

```
┌─── 오토 스케줄링 (인프라 레이어) ───┐        ┌─── 데일리 브리핑 (소비 레이어) ───┐
│                                │        │                                │
│  프롬프트 등록/수정/삭제             │        │  오토스케줄 결과 수집                │
│  실행 주기 설정                    │  ───→  │  시그널 탐지 & 종합                 │
│  스케줄러 실행 (정해진 시간)          │        │  인사이트 MD 리포트 생성              │
│  Raw 결과 저장                    │        │  추천 액션 제안                     │
│                                │        │                                │
└────────────────────────────────┘        └────────────────────────────────┘
```

이 구조의 핵심은, 오토 스케줄링은 "데일리 브리핑을 위해서만" 존재하는 것이 아니라는 점이다. 향후 프롬프트 실행 결과를 다른 소비처(예: 주간 리포트, 알림 트리거 등)에서도 활용할 수 있도록 인프라로 분리한다.

### 5.1. 오토 스케줄링 (인프라 레이어)

사용자가 등록한 프롬프트를 정해진 시간에 자동 실행하고, 결과를 저장하는 시스템이다.

**실행 흐름**

```
[사용자 프롬프트 등록] → [사전 검증 (1회 실행)] → [스케줄러 등록]
    → [정해진 시간에 프롬프트 개별 실행] → [Raw 결과 저장]
```

**프롬프트 실패 처리**

| 상황         | 처리 방식                                                                                                                 |
| ---------- | --------------------------------------------------------------------------------------------------------------------- |
| 등록 시 실패    | 사용자가 프롬프트를 실행해 보고 컨펌하는 사전 검증 단계 진행. 성공 시 "이런 형태로 결과가 나왔습니다. 주기적으로 호출하는 프롬프트에 추가할까요?" 확인. 실패 시 사용할 수 없음을 안내하고 다른 질의 제안 |
| 자동 실행 시 실패 | 최대 3회 재시도. 그래도 실패 시 프롬프트별 결과 확인 영역에 실패 상태 표기                                                                          |

### 5.2. 데일리 브리핑 (소비 레이어)

오토 스케줄링이 생성한 Raw 결과를 수집하여, 인사이트 리포트 에이전트가 하나의 종합 MD 리포트로 합성하는 시스템이다.

**실행 흐름**

```
[당일 오토스케줄 완료] → [Raw 결과 수집] → [인사이트 리포트 에이전트]
    → [시그널 탐지 & 종합] → [MD 리포트 + 추천 액션 생성]
        → [메인 화면 띠지 알림] → [사용자 확인]
```

**인사이트 리포트 생성 시 Input Data**

| Context         | 설명                            |
| --------------- | ----------------------------- |
| 프롬프트 응답값        | 등록된 프롬프트 각각의 실행 결과 (Raw Data) |
| 채용별 스냅샷 정보      | 현재 진행 중인 채용의 현황 데이터           |
| 사용자 리포트 생성 요구사항 | 사용자가 직접 입력한 리포트 포맷/관심 포인트     |
| 직전 데일리 브리핑 결과   | 전날 만들어진 데일리 브리핑 데이터           |

**리포트 결과 (Output)**

| 영역             | 설명                                                  | UI 형태                     |
| -------------- | --------------------------------------------------- | ------------------------- |
| 인사이트 리포트 본문    | 프롬프트 결과를 종합한 에이전트 작성 MD 리포트. 텍스트+볼드 수치+인라인 표를 자유 조합 | MD 렌더링 영역 (고정 컴포넌트 없음)    |
| 추천 액션 버튼       | 리포트 내용 기반으로 지금 해야 할 행동을 제안하는 딥링크 버튼                 | 필(pill) 형태 버튼 영역 (본문과 분리) |
| 프롬프트별 Raw Data | 각 프롬프트의 날 것의 결과값                                    | 아코디언 형태로 접/펼              |

## 6. 사용자 시나리오

### 6.1. 일상 사용: 매일 아침 브리핑 확인 → 후속 행동

(로그인 및 브리핑 알림 확인)

1. 채용 담당자 김가온이 오전 9시에 채용솔루션에 로그인한다.
2. 메인 화면 상단에 띠지 형태의 알림이 보인다: **"오늘의 데일리 브리핑이 준비됐어요"**
3. 김가온은 "오늘 채용 현황이 어떻게 됐는지 보자"라고 생각하며 띠지를 클릭한다.

(데일리 브리핑 페이지 진입)

1. 데일리 브리핑 전용 페이지로 이동한다.
2. 상단에 **인사이트 리포트**가 표시된다. 에이전트가 기본 템플릿 프롬프트(최근 일주일 지원자 수, 전형별 응시율, 접수 마감 임박 채용, 오늘 할 일 요약)의 실행 결과를 종합하여, **하나의 MD 리포트로 작성한** 내용이다. 고정된 카드나 stat-block이 아니라, 에이전트가 그날의 인사이트에 맞게 문단·볼드 수치·인라인 표를 자유롭게 조합한 형태다.
3. 오늘의 리포트에는 이런 내용이 담겨 있다:
  - 첫 문단: "이번 주 신규 지원자는 **47명**으로 전주(35명) 대비 **34%** 증가했습니다. 채용사이트(28명)와 링크드인(11명)이 주요 유입 채널입니다."
  - 인라인 표: 채용명별 신규 지원자 수, 전주 대비 증감, 누적 지원자
  - 강조 문단: "역량검사 전형의 응시율은 **72%**로 목표(80%) 대비 다소 낮습니다. 미응시자 34명 중 메일 미오픈 12명, 오픈 후 미응시 15명입니다."
4. 리포트 본문 아래에 **추천 액션 버튼 영역**이 분리되어 있다. 필(pill) 형태의 버튼이 나열된다:
  - `전형 안내 설정 화면으로 이동합니다. >`
  - `미응시자 독려 안내 설정으로 이동합니다. >`
5. 그 아래에 **프롬프트별 결과 영역**이 아코디언 형태로 나열되어 있다.
  - "최근 일주일간 지원자 수" — ✅ 성공
  - "전형별 응시율 현황" — ✅ 성공
  - "접수 마감 임박 채용" — ✅ 성공
  - "오늘 할 일 요약" — ✅ 성공

(인사이트에서 후속 행동으로 연결)

1. 김가온은 리포트를 읽으며 "역검 전형 안내를 아직 안 보냈구나"를 인지한다.
2. 리포트 아래 추천 액션 영역에서 **[전형 안내 설정 화면으로 이동합니다. >]** 버튼을 클릭한다.
3. 플로우 화면의 전형 안내 설정 페이지로 바로 이동한다.

(Raw Data 확인)

1. 김가온이 다시 브리핑 페이지로 돌아와 "최근 일주일간 지원자 수" 아코디언을 클릭한다.
2. 해당 프롬프트의 날 것의 실행 결과(에이전트 응답 원문)가 펼쳐진다. 실제 에이전트가 답변한 형태 그대로(텍스트+볼드+표+액션 링크) 표시된다.
3. 김가온은 "지원자가 전주보다 12명 늘었네? 어떤 채널에서 유입된 거지?"라고 생각한다.

(서브챗으로 추가 질의)

1. 김가온이 화면 우측의 **서브챗 열기** 버튼을 클릭한다.
2. 서브챗이 열리면서, 현재 데일리 브리핑 페이지의 요약 정보가 **자동으로 컨텍스트에 주입**된다.
3. 김가온이 "신규 지원자 12명의 유입 채널 분포를 알려줘"라고 입력한다.
4. 에이전트가 컨텍스트를 참고하여 바로 답변한다: "이번 주 신규 지원자 12명 중 채용사이트 7명, 링크드인 3명, 직접지원 2명입니다."
5. 김가온은 "채용사이트가 제일 많네. 이 정보도 매일 보면 좋겠다"라고 생각한다.

(완료)

1. 김가온은 필요한 정보를 모두 확인하고, 다른 업무로 이동한다.
2. 데일리 브리핑 페이지는 다음날 스케줄러가 돌 때까지 오늘의 결과를 유지한다.

### 6.2. 초기 설정: 프롬프트 등록 및 스케줄 세팅

(프롬프트 등록 시작)

1. 김가온이 데일리 브리핑 페이지에서 **[프롬프트 추가]** 버튼을 클릭한다.
2. 프롬프트 입력 화면이 열린다.
3. 김가온이 "매일 진행 중인 공채의 전형별 지원자 수를 알려줘"라고 입력한다.

(프롬프트 사전 검증)

1. 시스템이 입력한 프롬프트를 **즉시 1회 실행**하여 결과를 미리 보여준다.
2. 결과가 정상적으로 나온다.
3. 시스템이 "이런 형태로 결과가 나왔습니다. **이걸 주기적으로 호출하는 프롬프트에 추가할까요?**"라고 묻는다.
4. 김가온이 **[추가]** 버튼을 클릭한다.

(프롬프트 실패 시)

1. 만약 프롬프트 실행이 실패한다면, 시스템이 "이 프롬프트는 현재 사용할 수 없습니다"라고 공손히 안내한다.
2. 함께 **다른 질의 제안**을 표시한다: "이런 질의는 어떠세요? — '진행 중인 채용별 지원자 현황을 알려줘'"
3. 김가온이 제안된 질의를 선택하거나, 직접 수정하여 다시 시도할 수 있다.

(스케줄 설정)

1. 프롬프트가 등록되면, 스케줄 설정 화면이 표시된다.
2. 기본값은 **매일 오전 8:00**으로 설정되어 있다.
3. 김가온이 시간을 확인하고 [저장]을 클릭한다.
4. "프롬프트가 등록되었습니다. 내일 오전 8시에 첫 번째 결과를 확인할 수 있어요."라는 완료 메시지가 표시된다.

## 7. 데이터 구조

### 7.1. 등록 프롬프트

- **개념**: 사용자가 데일리 브리핑에 등록한 자동 실행 대상 프롬프트


| **컬럼명**         | **설명**     | **예시**                        |
| --------------- | ---------- | ----------------------------- |
| prompt_id       | 프롬프트 고유 ID | PROMPT_001                    |
| user_id         | 등록한 사용자 ID | USER_001                      |
| prompt_text     | 프롬프트 내용    | "매일 진행 중인 공채의 전형별 지원자 수를 알려줘" |
| schedule_time   | 실행 시간      | 08:00                         |
| status          | 상태         | active / paused / failed      |
| created_at      | 등록 일시      | 2026-03-02 09:15:00           |
| last_run_at     | 마지막 실행 일시  | 2026-03-02 08:00:00           |
| last_run_status | 마지막 실행 결과  | success / failed              |


- **기록 조건**: 사용자가 프롬프트를 등록(사전 검증 통과 후)할 때 생성
- **기록 시점**: 프롬프트 등록 시 생성, 실행 시마다 last_run 필드 업데이트
- **기록 단위**: 프롬프트 1개 = 레코드 1개

### 7.2. 실행 결과

- **개념**: 스케줄러가 프롬프트를 실행한 결과 (Raw Data)


| **컬럼명**        | **설명**        | **예시**                      |
| -------------- | ------------- | --------------------------- |
| result_id      | 결과 고유 ID      | RESULT_001                  |
| prompt_id      | 프롬프트 ID (FK)  | PROMPT_001                  |
| execution_date | 실행 일자         | 2026-03-02                  |
| raw_response   | 프롬프트 실행 결과 원문 | "전체 지원자 245명, 서류전형 120명..." |
| status         | 실행 상태         | success / failed            |
| retry_count    | 재시도 횟수        | 0                           |
| created_at     | 생성 일시         | 2026-03-02 08:00:15         |


- **기록 조건**: 스케줄러가 프롬프트를 실행할 때마다 생성
- **기록 시점**: 스케줄 실행 완료 시
- **기록 단위**: 프롬프트 1개 × 실행 1회 = 레코드 1개

### 7.3. 인사이트 리포트

- **개념**: 당일 실행된 프롬프트 결과를 종합 요약한 MD 형태의 리포트


| **컬럼명**        | **설명**            | **예시**                   |
| -------------- | ----------------- | ------------------------ |
| report_id      | 리포트 고유 ID         | REPORT_001               |
| user_id        | 사용자 ID            | USER_001                 |
| report_date    | 리포트 일자            | 2026-03-02               |
| summary_md     | 요약 리포트 (Markdown) | "## 오늘의 채용 브리핑\n..."     |
| source_results | 참조한 실행 결과 ID 목록   | [RESULT_001, RESULT_002] |
| created_at     | 생성 일시             | 2026-03-02 08:05:00      |


- **기록 조건**: 당일 모든 프롬프트 실행이 완료된 후 인사이트 리포트 에이전트가 생성
- **기록 시점**: 전체 프롬프트 실행 완료 직후
- **기록 단위**: 사용자 1명 × 일자 1일 = 리포트 1개

## 8. 의사결정 필요사항

- **프롬프트 등록 개수 상한**
  - 사용자당 등록 가능한 프롬프트 개수를 몇 개로 제한할 것인가?
  - 무제한 시 비용/성능 이슈 고려 필요
- **스케줄 실행 시간 커스터마이징**
  - 사용자별로 실행 시간을 자유롭게 설정 가능하게 할 것인가? 아니면 고정 시간(예: 매일 오전 8시)으로 할 것인가?
- **리포트 유지 기간**
  - 과거 리포트를 며칠치까지 보관/조회 가능하게 할 것인가?
- **서브챗 컨텍스트 범위**
  - 서브챗에 주입되는 정보가 인사이트 요약만인지, Raw Data까지 포함인지?
- **프롬프트 실패 시 사용자 알림**
  - 자동 실행 실패(3회 재시도 후에도 실패) 시 별도 알림(이메일, 앱 내 알림)을 보낼 것인가?

### 99.3. 메모

**기존 인사이트 카드와의 관계**

- 현재 메인 > 인사이트에 "지원서 제출 완료율" 등 고정 카드가 존재
- 데일리 브리핑은 이 고정 카드를 대체하는 것이 아니라, 사용자 맞춤 정보를 **추가**로 제공하는 기능
- 향후 고정 카드를 데일리 브리핑에 통합하거나, 공존시킬지는 별도 논의 필요

**오토 스케줄링 확장 가능성**

- 오토 스케줄링은 인프라 레이어로 분리되어 있으므로, 데일리 브리핑 외에도 다양한 소비처에서 활용 가능
- 예: 주간 리포트 생성, 특정 수치 임계값 초과 시 알림 트리거, 팀 단위 종합 대시보드 등
- 프롬프트 등록 시 사전 검증 플로우는 어떤 소비처에서든 재사용 가능

**솔루션 핵심 포인트**

- 사용자마다 보고 싶은 정보가 다르다는 전제. 통계 정보를 잘 처리할 수 있는 게 '오토 스케줄링'
- 오토 스케줄링이 Raw 데이터를 축적하고, 데일리 브리핑이 이를 소비하여 인사이트를 생성하는 두 레이어 구조
