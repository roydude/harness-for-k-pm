# 인터뷰 가이드

PM과의 핑퐁 대화에서 좋은 질문을 던지기 위한 가이드라인이다.

---

## 질문 원칙

### 한 번에 하나만

나쁜 예:
> "이 기능의 목적은 뭔가요? 타겟 유저는 누구고, 성공 지표는 어떻게 잡으셨어요?"

좋은 예:
> "이 기능을 만들려는 가장 큰 이유가 뭔가요?"

### 구체적으로 유도

PM이 추상적으로 답하면 구체화 질문을 던진다.

PM: "사용자 경험을 개선하려고요"
→ "구체적으로 어떤 경험이 안 좋은 건가요? 예를 들어 어떤 상황에서 불편함을 느끼나요?"

### 가정 후 확인

정보가 부족할 때, 가정을 세우고 확인하는 방식이 효율적이다.

> "말씀하신 내용으로 보면 [A라는 상황]에서 [B라는 동작]이 필요한 것 같은데, 맞나요?"

### "현상 → 시사점" 끌어내기

PM이 배경을 설명할 때 현상만 말하면, 전략적 시사점까지 끌어낸다.

PM: "사용자들이 기능을 잘 못 찾아요"
→ "그러면 기능을 직접 찾게 하기보다, 흐름 속에서 자연스럽게 안내하는 방향인 건가요?"

---

## Phase별 핵심 질문

### Phase 1: 기획 개요 파악

입력 정보가 아예 없을 때:
1. "어떤 제품/기능을 기획하고 계신가요? 제품명이 있다면 알려주세요."
2. "이 기능이 나오게 된 배경이 뭔가요? 어떤 문제나 기회가 있었나요?"
3. "이 제품/기능을 한 마디로 정의하면 뭐라고 하실 건가요?"

입력 정보가 일부 있을 때 (문서/메모 업로드):
1. "[업로드된 내용]을 보면 [X]가 핵심 개념인 것 같은데, 맞나요?"
2. "배경에서 [현상]은 이해했는데, 그래서 어떤 전략 방향을 잡으신 건가요?"

피그마 링크가 있을 때:
1. "피그마 디자인을 보니 [X 화면]이 있네요. 이 기능의 이름과 핵심 목적을 알려주세요."
2. "디자인에서 [Y 영역]이 보이는데, 이 부분의 의도를 설명해주실 수 있나요?"

### Phase 2: 목표와 가설 설정

1. "이 기능이 성공했다고 판단할 수 있는 핵심 지표가 있나요?"
2. "이 지표를 어떻게 측정하실 계획인가요?" (서베이? 로그 분석? A/B 테스트?)
3. "검증하고 싶은 가설이 있다면요? '~하면 ~할 것이다' 형태로 말씀해주세요."

지표가 모호할 때:
- PM: "만족도를 높이고 싶어요"
→ "만족도를 운영 만족(사용 편의)과 결과 만족(채용 품질) 중 어느 쪽으로 보시나요?"

### Phase 3: 핵심 경험 & 사용자 플로우

1. "이 기능에서 사용자가 겪는 가장 핵심적인 경험을 1~3가지로 요약하면요?"
2. "사용자가 이 기능을 처음 접하는 시점은 언제인가요?"
3. "전체적인 시나리오가 몇 가지나 있을 것 같으세요? 대분류로 나눠주세요."
4. "시나리오 중 우선순위가 가장 높은 건 어떤 건가요?"

핵심 경험 끌어내기:
- PM이 바로 디테일로 들어가려고 하면, 한 발 물러서 고수준 요약을 먼저 요청한다.
→ "세부 내용 전에, 이 기능이 사용자에게 제공하는 경험의 본질을 한두 문장으로 정리해볼까요?"

### Phase 4: 사용자 시나리오 구체화

각 시나리오마다:
1. "이 시나리오의 사용자는 누구인가요? 구체적인 상황도 알려주세요."
   (예: "채용 담당자 김가온이 오전 9시에 로그인하면...")
2. "첫 화면에서 사용자가 보는 것은 뭔가요?"
3. "[이전 단계] 다음에 사용자가 하는 행동은?"
4. "이 단계를 완료했다는 걸 시스템이 어떻게 아나요?"
5. "이 시나리오가 끝나면 다음으로 어디로 이어지나요?"

사용자의 내면 끌어내기:
→ "이걸 본 사용자가 어떤 생각을 하면서 클릭할 것 같으세요?"
→ "사용자가 '이게 뭐지?' 하고 혼란스러울 수 있는 지점이 있나요?"

### Phase 5: 기술 구조 & 데이터 & 커스텀 섹션

기술 구조:
1. "이 기능의 시스템 구조를 간단히 설명해주실 수 있나요?"
2. "다이어그램이나 구조도가 있나요?" (FigJam 링크 등)
3. "외부 시스템이나 API 의존성이 있나요?"

데이터:
1. "이 기능에서 다루는 핵심 데이터는 뭔가요?"
2. "이 데이터는 언제, 어떤 조건에서 생성/수정되나요?"
3. "데이터 필드와 예시를 알려주실 수 있나요?"

의사결정 필요사항:
1. "아직 결정 안 된 것이 있나요? 용어, 범위, 기술 선택 등"
2. "이 PRD를 리뷰하는 사람에게 의견을 받고 싶은 포인트가 있나요?"

커스텀 섹션:
1. "기본 구조 외에 추가하고 싶은 섹션이 있으신가요?"
2. "[프로젝트 특성]을 고려하면 [A, B 섹션]을 추가하는 게 좋을 것 같은데, 어떠세요?"

## 정보 소스별 대응

### PM이 구두로 설명할 때
- 핵심 키워드를 잡아서 구조화하며 확인
- "정리하면 [1, 2, 3]인데 맞나요?"

### 기존 문서/메모가 있을 때
- 문서를 먼저 읽고, 빠진 부분만 질문
- "문서에서 [X]는 잘 정리되어 있는데, [Y]에 대한 설명이 부족한 것 같아요."

### 피그마 디자인이 있을 때
- 디자인에서 읽을 수 있는 정보는 먼저 반영
- 디자인에서 읽을 수 없는 비즈니스 로직만 질문
- "화면에 [A 컴포넌트]가 있는데, 이 데이터의 갱신 주기는 어떻게 되나요?"

### 이전 PRD가 있을 때
- 변경점 중심으로 인터뷰 단축
- "이전 버전 대비 [X, Y]가 달라진 것 같은데, 다른 변경사항도 있나요?"

### PM이 메모/아이디어를 같이 전달할 때
- 메모에서 PRD 본문에 해당하는 내용과 "작업 노트"를 분리한다
- PRD 본문에 들어갈 내용은 해당 섹션에 배치하고
- 아이디어, 고민, 논의 내용은 부록(99.3 메모)에 보존한다

## 인터뷰 종료 시점

아래 조건이 충족되면 초안 생성(Phase 6)으로 넘어간다:

1. 기획 배경과 제품 정의가 명확하다
2. 목적/목표/가설이 측정 가능한 수준으로 정리됐다
3. 핵심 사용자 경험이 고수준으로 요약됐다
4. 우선순위 높은 시나리오가 내러티브 형태로 구체화됐다
5. (해당 시) 기술 구조/데이터 구조의 윤곽이 잡혔다
6. 미결 사항이 "의사결정 필요사항"에 명시됐다

위 조건 중 하나라도 부족하면, 해당 항목에 대해 추가 질문한다.
단, PM이 "나머지는 나중에 채울게요"라고 하면 현재 수집된 내용으로 초안을 먼저 생성한다.
