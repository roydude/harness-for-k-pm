# PRD Creator

PM이 에이전트와 대화하며 PRD를 구체화하는 스킬이다.
한 번에 완성본을 만들지 않고, **단계별 인터뷰 → 초안 생성 → 피드백 반영**의 사이클로 동작한다.

---

## 핵심 원칙

1. **PM의 머릿속을 구조화한다** — 모호한 아이디어를 질문으로 꺼내서 명확한 스펙으로 전환한다.
2. **한 번에 하나만 묻는다** — 질문 폭탄을 피하고, 각 단계에서 핵심 1~2개만 확인한다.
3. **피그마 연동은 선택적이다** — 피그마 링크가 있으면 스크린샷을 캡처해 PRD에 삽입하고, 없으면 텍스트만으로 완성한다.
4. **산출물은 Markdown이다** — 개발자가 바로 읽고 코멘트할 수 있는 형식.
5. **기획 메모도 PRD다** — PM이 올리는 작업 노트, 아이디어, 논의 내용도 부록에 담아 맥락을 보존한다.

---

## PRD 기본 섹션 구조

아래가 이 스킬이 생성하는 PRD의 기본 뼈대다.
인터뷰 과정에서 각 섹션에 필요한 정보를 수집한다.

| 번호 | 섹션 | 목적 |
|------|------|------|
| 1 | 기획 개요 | 제품명, 기획 배경(현상→전략), 제품 정의 |
| 2 | 목적 및 목표 | 핵심 목적 + 측정 가능한 핵심 지표(KPI) |
| 3 | 검증 가설 | "~하면 ~할 것이다" 형태의 검증 가능한 가설 |
| 4 | 핵심 사용자 경험 | 상세 시나리오 전, 사용자 경험의 본질을 고수준으로 요약 |
| 5 | 기술 구조 | 시스템 아키텍처, 데이터 흐름, 주요 컴포넌트 |
| 6 | 사용자 시나리오 | 페르소나 기반 내러티브 스토리 (가장 분량이 큰 섹션) |
| 7 | 데이터 구조 | 핵심 데이터 모델, 필드 정의, 기록 조건 |
| 8 | 의사결정 필요사항 | 아직 확정 안 된 이슈, 리뷰어에게 의견 요청 |
| 99 | 부록 | 관련 문서, 테스트 질의, 메모/아이디어, 변경 이력 |

PM이 원하면 섹션을 추가/제거/순서 변경할 수 있다.
추가 가능한 섹션은 `references/section-catalog.md`를 참조한다.

## 단계별 인터뷰 플로우

인터뷰는 아래 6단계로 진행한다. 각 단계에서 PM의 답변을 받은 후 다음으로 넘어간다.
PM이 이미 충분한 정보를 제공했다면(기존 문서 업로드, 상세한 설명 등) 해당 단계를 건너뛴다.
`references/interview-guide.md`에 Phase별 질문 예시가 있으니 참조한다.

### Phase 1: 기획 개요 파악

PM에게 다음을 확인한다:

- **이 제품/기능의 이름은?**
- **왜 만드는가?** (배경: 현상 → 시사점/전략 방향)
- **이 제품/기능을 한 마디로 정의하면?** (핵심 개념, 범위)
- **핵심 사용자는 누구인가?**

PM이 배경을 설명할 때 "현상"만 말하면, "그래서 어떤 전략이 필요한 건가요?"로 시사점까지 끌어낸다.

> 이 단계의 목표: 기획 개요(제품명/배경/정의)와 목적을 채울 정보를 얻는다.

### Phase 2: 목표와 가설 설정

PM에게 다음을 확인한다:

- **이 기능의 성공을 어떻게 측정할 건가요?** (핵심 지표)
- **검증하고 싶은 가설이 있나요?** ("~하면 ~할 것이다")

PM이 지표를 모호하게 말하면 구체화를 돕는다:
- "채용 만족도를 높인다" → "운영 만족과 결과 만족 중 어느 쪽인가요? 측정은 서베이로 하나요?"

> 이 단계의 목표: 목적 및 목표, 검증 가설 섹션을 채운다.

### Phase 3: 핵심 경험 & 사용자 플로우

PM에게 다음을 확인한다:

- **이 기능에서 사용자가 겪는 핵심 경험은?** (1~3가지 고수준 요약)
- **Happy Path 플로우는?** (시작 → 중간 → 끝)
- **어떤 시나리오들이 있는가?** (대분류, 우선순위)

핵심 경험은 상세 시나리오 이전에 "이 기능의 본질"을 정리하는 것이다.
예: "시스템이 사용자의 현재 상태를 보고 체크리스트를 추천한다"

피그마 링크가 있다면 이 단계에서 디자인 컨텍스트를 가져온다:
- `get_design_context` 또는 `get_metadata`로 노드 구조를 파악한다.
- `get_screenshot`으로 핵심 화면의 스크린샷을 캡처한다.
- 디자인에서 읽을 수 있는 정보(화면 구성, UI 요소)는 먼저 반영하고, 읽을 수 없는 비즈니스 로직만 질문한다.

> 이 단계의 목표: 핵심 사용자 경험 요약과 시나리오 대분류를 확정한다.

### Phase 4: 사용자 시나리오 구체화

Phase 3에서 잡은 시나리오 대분류를 하나씩 구체화한다.

각 시나리오마다 확인할 것:
- **페르소나**: 누가, 어떤 상황에서?
- **트리거**: 언제 이 시나리오가 시작되는가?
- **단계별 행동**: 사용자는 뭘 하고, 시스템은 뭘 보여주는가?
- **완료 조건**: 이 시나리오가 끝나는 시점은?
- **다음 연결**: 완료 후 어디로 이어지는가?

시나리오는 **내러티브 스토리텔링**으로 작성한다:
- 페르소나에 이름을 붙인다 (김가온, 이수진 등)
- 사용자의 내면 생각을 "따옴표"로 표현한다
- 시나리오 내 논리적 단계를 소제목(괄호)으로 구분한다
- 시스템 반응과 사용자 행동을 번갈아 기술한다

시나리오가 여러 개인 경우, 우선순위가 높은 것부터 하나씩 진행한다.
한 시나리오가 완성되면 PM에게 확인 후 다음 시나리오로 넘어간다.

> 이 단계의 목표: 모든 핵심 시나리오를 내러티브 형태로 구체화한다.

### Phase 5: 기술 구조 & 데이터 & 커스텀 섹션

PM에게 다음을 확인한다:

**기술 구조** (기능 특성에 따라):
- 시스템 간 데이터 흐름은? (아키텍처 다이어그램이 있다면?)
- 주요 컴포넌트/모듈은?
- 외부 시스템 의존성은?

**데이터 구조**:
- 핵심 데이터 모델은? (필드, 타입, 예시)
- 데이터의 기록 조건/시점/단위는?

**의사결정 필요사항**:
- 아직 확정 안 된 것이 있는가? (용어, 범위, 기술 선택 등)

**커스텀 섹션**:
- 기본 구조 외에 추가하고 싶은 섹션이 있는가?
- `references/section-catalog.md`에서 프로젝트 성격에 맞는 섹션을 제안한다.

> 이 단계의 목표: 기술 구조, 데이터 모델, 미결 이슈, 추가 섹션을 확정한다.

### Phase 6: 초안 생성 및 리뷰

위 단계에서 수집한 정보를 바탕으로 PRD 초안을 Markdown으로 생성한다.

`references/prd-template.md`를 읽고 그 구조에 맞춰 작성한다.

작성 원칙:
- **기획 배경**은 "현상 → 시사점" 구조로 쓴다 ("A인 상황이다, But B → 그래서 C 전략이 필요하다")
- **사용자 시나리오**가 PRD의 핵심이다. 가장 많은 분량을 할당한다.
- 개발자가 읽었을 때 "이렇게 구현하면 되겠다"고 느끼는 수준의 구체성
- 모호한 표현 대신 구체적인 조건과 값 사용
- 한국어를 기본으로 하되, 기술 용어는 영문 병기
- 부록에 PM이 공유한 관련 문서 링크, 테스트 질의, 메모/아이디어를 정리한다
- 부록 번호는 `99.`로 시작하여 본문과 시각적으로 구분한다

생성 후 PM에게 리뷰를 요청하고, 피드백을 반영하여 수정한다.
이 과정을 PM이 만족할 때까지 반복한다.

## 피그마 연동 가이드

피그마 링크가 제공된 경우 아래 절차로 디자인 정보를 PRD에 통합한다:

1. URL에서 `fileKey`와 `nodeId`를 추출한다
   - `https://figma.com/design/:fileKey/:fileName?node-id=1-2` → fileKey: `:fileKey`, nodeId: `1:2`
2. `get_design_context`로 해당 노드의 코드와 스크린샷을 가져온다
3. `get_metadata`로 하위 노드 구조를 파악하여 화면 구성을 이해한다
4. 필요시 `get_screenshot`으로 특정 노드의 스크린샷을 별도 캡처한다
5. PRD의 관련 섹션에 스크린샷 이미지를 삽입하고, 디자인 노트를 추가한다
6. 스크린샷과 함께 피그마 원본 링크도 병기한다 (스크린샷은 시점 고정, 링크는 최신 반영)

## 입력 소스별 대응

PM이 제공하는 정보의 형태에 따라 인터뷰를 유연하게 조절한다.

**구두 설명만 있을 때**: 전체 Phase를 순서대로 진행한다.

**기존 문서/메모가 있을 때**: 문서를 먼저 읽고, 이미 담긴 내용은 확인만 한다. 빠진 부분만 집중 질문하여 인터뷰를 단축한다.

**피그마 디자인이 있을 때**: 디자인에서 읽을 수 있는 정보(화면 구성, UI 요소, 흐름)는 먼저 반영한다. 디자인에서 읽을 수 없는 비즈니스 로직, 데이터 흐름, 엣지케이스만 질문한다.

**기존 PRD의 업데이트일 때**: 변경점 중심으로 인터뷰를 단축한다. "이전 버전 대비 달라진 점"을 먼저 확인한다.

## 출력 파일

최종 PRD는 Markdown 파일로 저장한다:
- 파일명 형식: `PRD_[기능명]_[YYYYMMDD].md`
- 저장 위치: 사용자가 선택한 작업 폴더

## 주의사항

- PM이 "그냥 빨리 만들어줘"라고 해도 최소한 Phase 1(기획 개요)과 Phase 3(핵심 경험)은 확인한다. 맥락 없이 만든 PRD는 쓸모가 없다.
- 인터뷰 중 PM의 답변이 모호하면, 가정을 세운 뒤 "이렇게 이해했는데 맞나요?"로 확인한다. 절대 혼자 결정하지 않는다.
- **의사결정 필요사항**은 빈 채로 두지 않는다. 인터뷰 중 "아직 정해지지 않은 것"이 보이면 적극적으로 기록한다. "확정된 것만 쓰는 문서"보다 "미결 사항이 보이는 문서"가 리뷰의 질을 높인다.
- 부록의 메모 섹션에 PM이 대화 중 언급한 아이디어, 논의 내용, 배경 맥락을 자유 형식으로 담는다. 이 내용은 나중에 PRD를 이해하는 데 중요한 맥락이 된다.
- 피그마 링크 없이도 PRD는 완전히 동작한다. 피그마 연동은 보너스이지 필수가 아니다.
